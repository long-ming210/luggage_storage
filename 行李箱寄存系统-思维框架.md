# 行李箱寄存系统 — 思维框架文档

> **视角**：独立开发者新手  
> **状态**：初期规划阶段，边做边学，逐步迭代  
> **核心原则**：先跑通最小可用版本（MVP），再逐步替换模拟模块为真实方案

---

## 一、项目一句话概括

一个 **Web 全栈** 行李箱寄存平台，用户可以在线查找寄存点、预约/扫码存取行李、在线支付；商户可以管理寄存点和订单、查看营收；智能柜硬件 **先用模拟方案跑通流程，后续替换为蓝牙协议**。

### 4.3 智能柜硬件对接（核心难点）

这是一个**从零开始的两阶段方案**：

#### 第一阶段：模拟方案（开发期）

```
前端请求 ──→ 后端 ──→ 模拟智能柜服务 (Mock Locker Service)
                         │
                         ├── 模拟格口状态（内存 Map 维护）
                         ├── 模拟开门指令（返回成功/失败）
                         ├── 模拟关门检测（返回门已关/超时）
                         └── 模拟故障注入（测试异常流程）
```

- 实现为一个 **Spring Boot 内部的 Service 层**，数据存在 Redis/内存中
- 对外暴露 RESTful API：`POST /api/locker/open`、`POST /api/locker/close`、`GET /api/locker/status/{slotId}`
- **设计关键**：接口契约和第二阶段真实蓝牙方案 **保持一致**，方便切换

#### 第二阶段：蓝牙方案（上线期）

```
Spring Boot ──→ 蓝牙通信服务 (独立模块) ──→ 蓝牙网关/适配器 ──→ 物理智能柜
                                              │
                                              ├── BLE 协议: GATT Profile
                                              ├── 开柜指令: Write Characteristic
                                              └── 状态回读: Read/Notify Characteristic
```

- 蓝牙通信服务作为 **独立 Spring Bean**，实现与模拟服务 **相同接口**
- 需要在服务器上插蓝牙适配器，或通过 **蓝牙网关设备**（网线/WiFi 转 BLE）连接柜子
- 需要考虑：蓝牙连接稳定性、断线重连、指令超时重试、并发连接数限制

> **🔧 待后续确定的技术细节：**
> - 蓝牙适配器选型（BlueZ 协议栈？特定品牌 SDK？）
> - 一台服务器能管理多少个蓝牙柜子？（并发连接上限）
> - 柜子硬件厂家的通信协议文档？

```

POST   /api/orders/{id}/store       # 扫码存件（触发开门）
POST   /api/orders/{id}/pickup      # 扫码取件（触发开门+结算）
```

### 6.4 支付 API（`/api/pay`）

```

POST   /api/pay/callback            # 真实支付回调（第二阶段）
```

### 6.5 商户端 API（`/api/merchant`）

```
POST   /api/merchant/login           # 商户登录

POST   /api/merchant/points          # 添加寄存点
PUT    /api/merchant/points/{id}     # 编辑寄存点

GET    /api/merchant/orders          # 订单管理列表
GET    /api/merchant/revenue         # 营收统计
```

### 6.6 智能柜 API（`/api/locker`）— 内部调用

```
POST   /api/locker/open/{slotId}     # 开柜指令
POST   /api/locker/close/{slotId}    # 关门检测
GET    /api/locker/status/{slotId}   # 格口状态查询
GET    /api/locker/point/{pointId}   # 某寄存点所有格口状态
```

## 七、智能柜通信模块设计（重点）

### 7.1 接口契约（关键设计）

```java
// 核心接口 — 模拟和蓝牙实现同一个接口，方便切换
public interface LockerCommunicationService {
    
    /** 发送开柜指令 */
    LockerResult openSlot(String deviceId, String slotNumber);
    
    /** 查询格口状态 */
    SlotStatus getSlotStatus(String deviceId, String slotNumber);
    
    /** 关门检测（主动查询或被回调） */
    boolean isDoorClosed(String deviceId, String slotNumber);
    
    /** 健康检查 */
    boolean healthCheck(String deviceId);
}
```

### 7.2 模拟实现（第一阶段）

```java
@Service
@Profile("dev")  // 开发环境启用
public class MockLockerService implements LockerCommunicationService {
    
    // 用 ConcurrentHashMap + Redis 维护模拟柜状态
    private final Map<String, SlotStatus> mockStatusMap = new ConcurrentHashMap<>();
    
    @Override
    public LockerResult openSlot(String deviceId, String slotNumber) {
        // 模拟：90%成功率，随机延迟100-500ms
        String key = deviceId + ":" + slotNumber;
        mockStatusMap.put(key, SlotStatus.OPEN);
        return LockerResult.success("模拟开门成功");
    }
    // ...
}
```

## 十一、写给新手的备注

1. **不要一上来就做蓝牙**。先用模拟把业务流程跑通，蓝牙是最后一步。
2. **数据库设计不要过度**。上面的表结构是一个起点，边做边改。
3. **一个功能一个分支**。用 Git 管好代码，别把所有功能写在一个巨大提交里。
4. **先让一个用户能存能取**，再考虑并发、性能、高可用。
5. **遇到问题先搜**。Spring Boot + Vue 的博客教程非常多，80%的坑别人都踩过。
6. **配置文件分环境**。`application-dev.yml`（本地开发）和 `application-prod.yml`（服务器）分开，不要混用。
